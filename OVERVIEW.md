# Обзор проекта: Сервис для обмена книгами между пользователями (буккроссинг)

## 📋 Описание проекта

Данное веб-приложение представляет собой сервис буккроссинга — обмена книгами между пользователями по принципу «прочитал — отдай другому». Система разработана как Single Page Application (SPA) без использования `react-router`, что обеспечивает быструю навигацию между разделами без перезагрузки страницы.

**Основная цель проекта**: автоматизация работы книгообменного пункта — учёт книг в обороте, выдача книг читателям и приём возвратов через простой веб-интерфейс.

## 💡 Идея и концепция

Буккроссинг (книговорот) — это движение по безвозмездному обмену книгами. В проекте реализована модель **книгообменного пункта** (как в библиотеке или коворкинге):

- Книги в общий оборот вводит **администратор** (смотритель пункта).
- **Читатель** берёт свободную книгу («Взять себе») и затем возвращает её в оборот («Вернуть»), чтобы книга продолжила путь к другим читателям.
- Каждое действие читателя оформляется как **заявка**, которую подтверждает администратор.

### Как реализованы пользователи без «настоящей» регистрации

Чтобы не усложнять проект полноценной регистрацией (e-mail, пароли, их хранение и защита), но при этом иметь полноценных пользователей-читателей, применена доменно-корректная модель **читательских билетов**:

- Регистрация читателей **вынесена за пределы системы** — читательские билеты выдаёт администратор (как в реальной библиотеке).
- Читатель **идентифицируется по номеру читательского билета** (вводится один раз, сохраняется в `localStorage` — аналог «вставил карту»). Пароли у читателей отсутствуют.
- Это снимает вопросы информационной безопасности (нет хранения и хеширования паролей пользователей) и при этом полностью соответствует предметной области буккроссинга.

## 🎯 Основные возможности системы

### Для всех пользователей (включая гостей):

1. **Каталог книг** — просмотр всех книг в обороте, поиск по названию и автору, фильтрация по жанру и статусу.
2. **Список читателей** — участники книгообменного пункта.

### Для читателей (вход по читательскому билету):

1. **Взять книгу** — оформление заявки на свободную книгу (кнопка «Взять себе»).
2. **Вернуть книгу** — оформление заявки на возврат книги, которая находится на руках.
3. **Мои заявки** — история собственных заявок и их статусы.

### Для администратора (вход по паролю):

1. **Управление книгами** (CRUD) — добавление, редактирование, деактивация книг.
2. **Управление жанрами** (CRUD).
3. **Управление читателями** (CRUD) — выдача и редактирование читательских билетов.
4. **Обработка заявок** — одобрение/отклонение заявок на выдачу и возврат книг.

## 💡 Почему React, а не Laravel (PHP)?

Проект реализован на React вместо традиционного стека Laravel + PHP по двум ключевым причинам:

1. **Встроенная безопасность после сборки (обфускация кода)**
   При выполнении команды `npm run build` весь исходный код проекта проходит через процесс **минификации и обфускации**. Это означает, что итоговые JavaScript-файлы становятся практически нечитаемыми для человека — переменные переименовываются в короткие бессмысленные имена, убираются все комментарии, код сжимается. В результате, даже если кто-то получит доступ к файлам на хостинге, восстановить исходную логику приложения будет крайне затруднительно. В Laravel исходный PHP-код остаётся на сервере в читаемом виде.

2. **Бесплатный хостинг на статичных платформах**
   После сборки React-приложение превращается в набор обычных статичных файлов (HTML, CSS, JS), которые можно разместить на **любом бесплатном статичном хостинге**, например:
   - **Vercel** — деплой в один клик, автоматический CI/CD
   - **GitHub Pages** — бесплатный хостинг прямо из репозитория

   Laravel и PHP требуют серверный хостинг с поддержкой PHP и базы данных, что практически всегда **платно**, либо доступно только с жёсткими ограничениями и сложной настройкой.

## 🔐 Система ролей и прав доступа

1. **Гость** (роль по умолчанию)
   - Просмотр каталога книг и списка читателей
   - Без возможности брать книги и оформлять заявки

2. **Читатель (member)** — вход по читательскому билету
   - Все права гостя
   - Оформление заявки «Взять себе» на свободную книгу
   - Оформление заявки на возврат книги с рук
   - Просмотр своих заявок

3. **Администратор (admin)** — вход по паролю
   - Управление книгами, жанрами и читательскими билетами (CRUD)
   - Одобрение и отклонение заявок
   - Доступ к разделу «Администрирование»

**Учетные данные** находятся в файле `CREDENTIALS.md`

## 🗄 Структура данных

### Основные сущности:

1. **Genre (Жанр)** — id, name, description, isActive
2. **Member (Читатель)** — id, cardNumber (читательский билет), firstName, lastName, email, phone, joinedDate, isActive
3. **Book (Книга)** — id, title, author, genreId, condition (new/good/used/worn), description, status (available/reserved/taken), holderId, isActive
4. **BookRequest (Заявка)** — id, bookId, memberId, type (take/return), status (pending/approved/rejected), createdAt, resolvedAt

### Жизненный цикл книги

```
available (свободна)
   │  читатель: «Взять себе» → заявка take (pending)
   ▼
reserved (бронь)
   │  администратор одобрил → taken; администратор отклонил → available
   ▼
taken (на руках)
   │  читатель: «Вернуть» → заявка return (pending)
   │  администратор подтвердил → available
   ▼
available (снова в обороте)
```

## 💻 Как работать с проектом

### Навигация

Система использует кастомную навигацию без `react-router` через `NavigationStore`:

- `navigationStore.navigate('books')` — переход на страницу
- Текущая страница хранится в `navigationStore.currentPage`
- Навигация автоматически проверяет права доступа

### Работа с данными

Все операции выполняются через `DataStore`:

```typescript
import { dataStore } from '@/store';

// Загрузка данных
dataStore.loadAllData();

// Читатель берёт книгу (создаёт заявку)
dataStore.requestBook(bookId);

// Читатель возвращает книгу
dataStore.requestReturn(bookId);

// Администратор обрабатывает заявку
dataStore.approveRequest(requestId);
dataStore.rejectRequest(requestId);
```

### Работа с авторизацией

```typescript
import { authStore } from '@/store';

// Вход администратора
authStore.loginAsAdmin('admin2026');

// Вход читателя по номеру билета
authStore.loginAsReader('BC-001');

// Проверка роли
if (authStore.isAdmin) { /* ... */ }
if (authStore.isMember) { /* ... */ }

authStore.logout();
```

### Стилизация

Для изменения цветовой схемы всего сайта достаточно изменить файл `src/styles/globalColors.scss`.

## 📚 Что желательно понимать для защиты диплома

### Теоретическая часть:

1. **SPA (Single Page Application)**
   - Что такое SPA и его преимущества? Ответ: веб-приложение, которое загружается один раз, а переходы между разделами происходят без перезагрузки. Преимущества: скорость, плавность, меньше нагрузки на сеть, лучший UX.
   - Почему без react-router? Ответ: бесплатный хостинг GitHub Pages не позволяет удобно обслуживать несколько маршрутов; для данного объёма проще и надёжнее кастомная навигация через MobX-store со switch-case в `App.tsx`.

2. **Почему нет регистрации и паролей у читателей? (ключевой вопрос)**
   - Ответ: реализована модель книгообменного пункта. Читательские билеты выдаёт администратор (как в библиотеке), а читатель идентифицируется по номеру билета. Это доменно-корректное решение, которое упрощает доступ и снимает вопросы безопасности паролей: персональные данные сверх необходимого не хранятся, паролей читателей в системе нет.

3. **State Management (MobX)**
   - Зачем нужен MobX? Ответ: для централизованного хранения состояния (книги, читатели, заявки) и реактивного обновления интерфейса без «прокидывания» данных через множество компонентов.
   - Как организованы stores? Ответ: AuthStore (вход и роли), DataStore (данные и бизнес-операции), NavigationStore (навигация), UIStore (модалки и уведомления).

4. **TypeScript**
   - Преимущества типизации? Ответ: ошибки находятся на этапе написания кода; компилятор проверяет типы и предотвращает обращения к несуществующим полям и неправильную передачу данных.

5. **Firebase Realtime Database**
   - Что такое Firebase RTDB? Ответ: облачная NoSQL база данных, хранящая данные в формате JSON и синхронизирующая их в реальном времени.
   - Как организована структура данных? Ответ: разделы `genres`, `members`, `books`, `requests`; каждая запись имеет уникальный идентификатор.
   - Особенность RTDB: база **не принимает значения `undefined`**. Поэтому перед записью данные пропускаются через функцию `sanitizeForFirebase` в `src/firebase.ts`, которая удаляет undefined-поля.
   - Преимущества/недостатки NoSQL? Ответ: гибкая структура, простота и скорость; но нет строгих связей и сложных запросов — целостность контролируется в коде.

6. **Какую роль играет обфускация React в продакшене?**
   - Реализовано ли хеширование пароля администратора? Ответ: нет, и в этом нет необходимости: production-сборка минифицируется и обфусцируется, найти конкретную строку-пароль среди тысяч случайных символов практически невозможно.

### Практическая часть (что можно продемонстрировать):

1. **Работа с ролями:** вход как читатель (по билету) и как администратор (по паролю), разница в правах.
2. **Сценарий буккроссинга:** читатель берёт свободную книгу → администратор одобряет → книга «на руках» → читатель возвращает → администратор подтверждает → книга снова свободна.
3. **CRUD:** администратор добавляет книгу/жанр/читательский билет, редактирует и деактивирует.
4. **Фильтрация и поиск:** поиск книг по названию/автору, фильтр по жанру и статусу.
5. **Адаптивность:** мобильное меню и адаптивная вёрстка.

### Возможные вопросы:

1. **«Почему нет полноценной регистрации?»** — модель книгообменного пункта с выдачей читательских билетов администратором; это упрощает доступ и снимает риски хранения паролей.
2. **«Как обеспечивается разграничение доступа?»** — проверка прав на уровне навигации (NavigationStore) и на уровне операций (DataStore + AuthStore.permissions).
3. **«Как можно расширить функционал?»** — рейтинги и отзывы о книгах, история перемещений книги, уведомления, QR-коды на билетах, статистика и отчёты.

## 🔍 Краткий экскурс в проект для защиты

Если вы знакомы с HTML, CSS и JavaScript, но не работали с React и TypeScript — этот раздел поможет вам быстро понять, как устроен проект и что где находится.

### Что такое React?

React — это JavaScript-библиотека для создания пользовательских интерфейсов. Вместо обычного HTML здесь используется **JSX** (JavaScript XML) — расширение синтаксиса JS, которое позволяет писать разметку прямо в JavaScript-коде. Файлы с расширением `.tsx` — это то же самое, но с поддержкой TypeScript.

**Простая аналогия**: если в обычном веб-сайте у вас есть `index.html` + `script.js`, то в React весь HTML и JS объединены в **компоненты** — функции, которые возвращают разметку.

### Что такое TypeScript?

TypeScript — это JavaScript с **типизацией**. Он добавляет возможность указывать типы переменных, параметров функций и возвращаемых значений. Это помогает находить ошибки ещё до запуска кода. После сборки TypeScript превращается в обычный JavaScript.

**Простая аналогия**: TypeScript — это как проверка орфографии для кода. Файлы `types/*.types.ts` описывают «форму» данных — какие поля есть у объекта и какого они типа. Это похоже на описание колонок в таблице базы данных, но на уровне кода.

### Как устроен проект?

#### 1. Точка входа — `main.tsx` и `App.tsx`

Всё начинается с файла `main.tsx`, который рендерит (отображает) компонент `App` на странице. `App.tsx` — это главный компонент приложения. При загрузке он вызывает `dataStore.loadAllData()` — загружает все данные из Firebase, оборачивает содержимое в `MainLayout` (шапка + боковое меню + контент) и содержит `PageRouter` — компонент, который решает, какую страницу показать.

#### 2. Навигация (без react-router)

В проекте **нет** библиотеки `react-router`. Навигация реализована вручную через **MobX store**:

- `NavigationStore` хранит переменную `currentPage` (например, `'home'`, `'books'`, `'requests'`, `'members'`, `'admin'`)
- Компонент `PageRouter` в `App.tsx` через `switch/case` проверяет `currentPage` и рендерит нужную страницу:
  - `'home'` → `HomePage` (дашборд)
  - `'books'` → `BooksPage` (каталог книг)
  - `'requests'` → `RequestsPage` (заявки на выдачу и возврат)
  - `'members'` → `MembersPage` (читатели)
  - `'admin'` → `AdminPage` (администрирование)
- Когда пользователь нажимает на пункт в боковом меню (`Sidebar`), вызывается `navigationStore.navigate('books')`, что меняет `currentPage`, и React автоматически перерисовывает нужную страницу

#### 3. Управление состоянием — MobX

**MobX** — это библиотека для управления состоянием приложения (аналог «глобальных переменных», доступных из любого компонента). В проекте 4 store:

- **AuthStore** (`store/AuthStore.ts`) — авторизация и роли пользователей
- **DataStore** (`store/DataStore.ts`) — все данные приложения (книги, читатели, жанры, заявки) и CRUD-операции
- **NavigationStore** (`store/NavigationStore.ts`) — текущая страница и навигация
- **UIStore** (`store/UIStore.ts`) — управление модальными окнами, уведомлениями, темой

Компоненты оборачиваются в `observer()` из `mobx-react-lite` — это позволяет React автоматически перерисовывать компонент при изменении данных в store.

#### 4. Авторизация и роли

`AuthStore` управляет системой ролей:

- **guest** (Гость) — роль по умолчанию, только просмотр каталога книг и списка читателей
- **member** (Читатель) — вход **по номеру читательского билета без пароля**; может брать и возвращать книги (оформлять заявки)
- **admin** (Администратор) — вход по паролю; полный доступ: управление книгами, жанрами, читательскими билетами и обработка заявок
- Пароль администратора захардкожен в коде (это учебный проект, не production); у читателей пароля нет — они идентифицируются по билету, который выдаёт администратор
- Сессия сохраняется в `localStorage` на 24 часа
- Проверка прав: объект `ROLE_PERMISSIONS` определяет, что может делать каждая роль
- При попытке перейти на защищённую страницу без авторизации открывается модальное окно входа

#### 5. Работа с данными — Firebase

`firebase.ts` инициализирует подключение к **Firebase Realtime Database** (облачная NoSQL база данных от Google). Класс `FirebaseService` предоставляет методы:

- `getData(path)` — получить данные
- `setData(path, data)` — записать данные
- `updateData(path, updates)` — обновить данные

`DataStore` использует эти методы для CRUD-операций с книгами, читателями, жанрами и заявками. Каждая сущность хранится в массиве, при создании/редактировании данные сначала отправляются в Firebase, а затем обновляются локально.

> **Важная деталь:** Firebase Realtime Database **не принимает значения `undefined`**. Поэтому перед записью данные пропускаются через функцию `sanitizeForFirebase` в `src/firebase.ts`, которая рекурсивно удаляет undefined-поля — иначе запись завершилась бы ошибкой.

#### 6. Компоненты и страницы

**Компоненты** (`src/components/`):
- `UI/` — переиспользуемые элементы: Button, Input, Modal, Table, Select, Badge, Card
- `Layout/` — структура страницы: Header (шапка), Sidebar (боковое меню), MainLayout (общий каркас)
- `LoginModal/` — модальное окно входа (вкладки «Читатель» по билету и «Администратор» по паролю)
- `ConfirmModal/` — модальное окно подтверждения действий (например, удаления)

**Страницы** (`src/pages/`):
- `HomePage/` — дашборд со статистикой: всего книг, свободных, на руках, читателей и заявок в ожидании
- `BooksPage/` — каталог книг с поиском по названию/автору и фильтрацией по жанру и статусу; кнопки «Взять себе» / «Вернуть» для читателя
- `RequestsPage/` — заявки на выдачу и возврат: читатель видит свои, администратор обрабатывает все («Одобрить» / «Отклонить»)
- `MembersPage/` — список читателей книгообменного пункта
- `AdminPage/` — панель администрирования с CRUD-операциями для книг, жанров и читательских билетов

Каждая страница — это папка с тремя файлами:
- `[PageName].tsx` — React-компонент с логикой и разметкой
- `[PageName].module.scss` — стили (CSS-модули + SCSS)
- `index.ts` — реэкспорт компонента для удобного импорта

#### 7. Стили

Проект использует **SCSS** (расширение CSS) с модульным подходом:
- `*.module.scss` — стили, привязанные к конкретному компоненту (не конфликтуют с другими)
- `styles/globalColors.scss` — цветовая палитра проекта (изумрудно-бирюзовая тема с коралловым акцентом)
- `styles/variables.scss` — CSS-переменные (размеры, отступы, шрифты)
- `styles/mixins.scss` — повторно используемые блоки стилей
- `styles/global.scss` — глобальные стили для всего приложения
